STARTUPGROW - COMPLETE PROJECT DETAILS
======================================

Project Name: STARTUPGROW
Project Type: Web-based digital solutions platform
Target Users: Indian startups, early-stage founders, and the StartupGrow admin team
Technology Stack: Next.js, React, Tailwind CSS, JavaScript, Node.js, Express.js, MongoDB, Mongoose
Team Members: Yadunand Pavithran, Nashith Nihal C, Gokul Krishna


1. PROJECT OVERVIEW
-------------------

STARTUPGROW is a web-based digital solutions platform created to help Indian startups build and improve their online presence. The platform focuses on services that a new startup usually needs during its early growth stage, such as website development, SEO, PPC advertising, social media marketing, content marketing, email marketing, and branding support.

The main purpose of STARTUPGROW is to bring these services into one simple platform. Instead of a founder searching for different freelancers, agencies, and tools, the founder can visit STARTUPGROW, understand the available services, submit a project request, and wait for the team to review it through the admin dashboard.

In simple words, STARTUPGROW connects startup founders with digital growth services through a modern website and a basic request management system.


2. IDEA BEHIND STARTUPGROW
--------------------------

The idea came from a common problem faced by many early-stage startups in India. Many founders have good business ideas, but they struggle to create a professional online presence because they may not have technical knowledge, design skills, marketing experience, or enough budget to hire large agencies.

Some common problems are:

- Startups do not know how to create a professional website.
- Many startups have poor visibility on search engines.
- Founders may not know how to run social media campaigns.
- Digital agencies can be expensive for new businesses.
- Startup teams may not have time to contact many service providers separately.
- There is no simple single platform for requesting digital growth support.

STARTUPGROW was planned as a solution to this problem. The platform gives startups one place to explore digital services and send a structured request. The admin team can then view those requests and manage the status of each inquiry.


3. PROJECT OBJECTIVES
---------------------

The main objective of STARTUPGROW is to design and develop a web-based platform that helps Indian startups grow online.

Specific objectives include:

- Build a professional and responsive website for the platform.
- Display the main digital services clearly.
- Allow startup founders to submit service requests.
- Store submitted requests in a backend system.
- Provide an admin dashboard for request tracking.
- Support status management such as New, Contacted, In progress, and Closed.
- Keep the platform simple enough for academic demonstration and scalable enough for future improvements.


4. SERVICES PROVIDED BY STARTUPGROW
-----------------------------------

STARTUPGROW focuses on six core services:

1. Search Engine Optimisation
   This service helps improve a startup website's ranking in search engines. It includes keyword planning, content optimisation, technical SEO, and organic traffic improvement.

2. Pay-Per-Click Advertising
   This service helps startups run paid advertisements on platforms such as Google, Instagram, and other digital ad networks. The goal is to generate leads and visibility quickly.

3. Social Media Marketing
   This service helps startups manage their social media presence through account setup, content planning, posts, reels, campaign management, and audience engagement.

4. Content Marketing
   This service focuses on blogs, website copy, videos, campaign messages, and educational posts that help attract and convert users.

5. Email Marketing
   This service helps startups send promotional emails, newsletters, welcome emails, follow-up messages, and campaign sequences.

6. Web Design and Development
   This service helps startups build responsive websites, landing pages, contact forms, and digital interfaces that can convert visitors into customers.


5. HOW THE PROJECT WAS MADE
---------------------------

STARTUPGROW was developed in stages. The first step was identifying the problem faced by startups. After that, the project was divided into modules and implemented using a modern JavaScript stack.

The development process followed this flow:

1. Requirement Analysis
   The project requirements were identified based on what early-stage startups need. The main requirements were a public website, service information, contact/request form, backend API, database storage, and an admin dashboard.

2. UI and Page Planning
   The public website pages were planned first. The website needed pages such as Home, About, Services, Works, Team, Career, FAQ, Blog, and Contact. The design was based on a modern agency-style layout with animations, images, and responsive sections.

3. Frontend Development
   The frontend was built using Next.js and React. Static assets such as CSS files, images, icons, fonts, and JavaScript animation files are stored in the public folder. The public pages use Arolax-style template assets and custom project content related to STARTUPGROW.

4. Contact Form Development
   A React form component was created to collect startup service request details. The form collects startup name, founder name, email, phone, city, stage, required service, budget range, and project goal.

5. Backend API Development
   The backend logic was created using Node.js and Express.js. The backend validates form data, stores service requests, stores contact leads, lists records, and updates request status.

6. Database Integration
   MongoDB was used as the database, and Mongoose was used to define schemas and interact with collections. The project also includes a temporary in-memory fallback if MongoDB is not configured or unavailable.

7. Dashboard Development
   A dashboard named SHIELD was created for admin use. It allows the admin team to view service requests, view contact leads, refresh data, and update statuses.

8. Deployment Preparation
   The project includes same-origin Next.js API routes, so it can work on platforms like Vercel without always depending on a separate Express server. It can also run with a standalone Express backend if needed.

8. CONTACT FORM AND SERVICE REQUEST FLOW
----------------------------------------

The service request form is handled by the QuoteForm component located at components/QuoteForm.jsx.

The form collects:

- Startup name
- Founder name
- Email address
- Phone number
- City
- Startup stage
- Primary service
- Budget range
- Project goal/message

Default values include:

- Startup stage: Idea stage
- Service: Web Design & Development
- Budget range: Below Rs. 25,000

Form validation on the frontend checks that important fields are filled before allowing submission. When the user submits the form, QuoteForm calls createServiceRequest from lib/api.js. That function sends a POST request to /api/requests.

After a successful submission:

- The form is reset.
- A success message is shown.
- The request becomes available in the SHIELD dashboard.

If there is an error:

- An error message is shown to the user.
- The form is not silently ignored.


9. BACKEND DETAILS
------------------

The backend is built using Node.js and Express.js. It is located inside the server folder.

Important backend files:

1. server/index.js
   This is the standalone Express server. It configures environment variables, CORS, JSON parsing, API routes, error handling, database connection, and server startup.

2. server/apiCore.js
   This file contains reusable API logic. The same logic is used by the Next.js API routes. This prevents duplication and allows the app to work both with a standalone Express server and with same-origin Next.js API routes.

3. server/db.js
   This file manages the MongoDB connection using Mongoose. It reads MONGODB_URI from the environment and connects to MongoDB. If MongoDB is unavailable, it allows the API to continue using in-memory storage.

4. server/models/ServiceRequest.js
   This file defines the Mongoose schema for service requests.

5. server/models/ContactLead.js
   This file defines the Mongoose schema for general contact leads.


10. DATABASE DETAILS
--------------------

MongoDB is used as the main database. Mongoose is used to define schemas and communicate with MongoDB.

ServiceRequest fields:

- startupName
- founderName
- email
- phone
- city
- stage
- service
- budget
- message
- status
- createdAt
- updatedAt

ContactLead fields:

- name
- email
- phone
- subject
- service
- message
- status
- createdAt
- updatedAt

The status field supports these values:

- New
- Contacted
- In progress
- Closed

The default status for a new request or lead is New.

If MONGODB_URI is missing or MongoDB cannot connect, the project uses temporary in-memory arrays. This is useful for demo purposes, but in-memory data is lost when the server restarts.


11. API ENDPOINTS
-----------------

The project supports these API endpoints:

1. GET /api/health
   Checks whether the API is running and shows whether the database mode is MongoDB or memory.

2. GET /api/services
   Returns the list of available STARTUPGROW services.

3. POST /api/requests
   Creates a new service request from the public request form.

4. GET /api/requests
   Lists recent service requests for the dashboard.

5. PATCH /api/requests/:id/status
   Updates the status of a service request.

6. POST /api/leads
   Creates a general contact lead.

7. GET /api/leads
   Lists recent contact leads.

8. PATCH /api/leads/:id/status
   Updates the status of a contact lead.

The API validates required fields, checks email format, checks whether the selected service is valid, and returns suitable success or error responses.


12. NEXT.JS API ROUTES
----------------------

The app folder contains Next.js API routes inside app/api. These routes import functions from server/apiCore.js.

This is useful because the project can run API logic directly inside the Next.js app. For example:

- app/api/requests/route.js handles GET and POST for service requests.
- app/api/leads/route.js handles GET and POST for contact leads.
- app/api/requests/[id]/status/route.js handles status updates.
- app/api/leads/[id]/status/route.js handles lead status updates.

The route handlers use Node.js runtime and force dynamic behavior so request data is always fresh.


13. API HELPER ON FRONTEND
--------------------------

The file lib/api.js contains helper functions used by the frontend.

Important functions:

- createServiceRequest(data)
- fetchServiceRequests()
- updateRequestStatus(id, status)
- createContactLead(data)
- fetchContactLeads()
- updateLeadStatus(id, status)

The helper supports NEXT_PUBLIC_API_URL. If NEXT_PUBLIC_API_URL is provided, the frontend first tries the external API URL. If the external API route is missing or unavailable due to CORS/network errors, it falls back to the same-origin /api routes inside Next.js.

This makes the project more flexible for local development and deployment.


14. DASHBOARD DETAILS - SHIELD ADMIN PORTAL
-------------------------------------------

The dashboard is called SHIELD. It is located at /shield.

The dashboard is designed as a dark admin interface with a sidebar and main content area. It is made using React, Tailwind CSS, and lucide-react icons.

Main dashboard tabs:

1. Overview
   Shows service requests submitted by startup founders.

2. Test Request
   Provides a test version of the request form inside the admin dashboard. This helps simulate a public request and confirm that the database connection works.

3. Contact Leads
   Shows general contact leads submitted through the system.

Dashboard features:

- Displays total number of service requests.
- Displays active requests, excluding Closed requests.
- Shows a notification box when there are new service requests.
- Allows manual refresh of data.
- Automatically refreshes service requests every 15 seconds.
- Displays request details such as startup name, city, stage, founder name, email, service, and status.
- Allows the admin to update request status using a dropdown.
- Displays color-coded status badges.


15. HOW THE DASHBOARD WORKS
---------------------------

The dashboard workflow is:

1. A founder submits a request from the public Contact page.
2. The QuoteForm component sends the data to /api/requests.
3. The backend validates the data.
4. If MongoDB is connected, the request is saved in the ServiceRequest collection.
5. If MongoDB is not connected, the request is saved temporarily in memory.
6. The SHIELD dashboard calls fetchServiceRequests().
7. fetchServiceRequests() sends GET /api/requests.
8. The backend returns the latest requests sorted by createdAt descending.
9. DashboardView stores the result in React state.
10. The admin sees the request in the table.
11. The admin changes the status using the dropdown.
12. DashboardView calls updateRequestStatus(id, status).
13. The backend updates the request status in MongoDB or memory.
14. The updated request is returned and displayed immediately in the dashboard.

The same idea is used for Contact Leads:

1. LeadsView calls fetchContactLeads().
2. It displays lead name, subject, email, phone, service, and status.
3. Admin can update lead status.
4. The backend stores the updated status.


16. STATUS MANAGEMENT
---------------------

The project uses four statuses:

1. New
   A newly submitted request or lead that has not yet been handled.

2. Contacted
   The team has contacted the founder or lead.

3. In progress
   The request is being discussed, reviewed, or worked on.

4. Closed
   The request or lead is completed, rejected, or no longer active.

The dashboard uses status badges with different colors and icons so the admin can quickly understand the current condition of each request.

The system can be divided into four main modules:

2. Service Module

18. SYSTEM ARCHITECTURE
-----------------------

STARTUPGROW follows a client-server architecture.

Flow:

1. User opens the Next.js frontend in the browser.
2. User explores public pages and submits a request.
3. React form sends data to the API.
4. API validates the request.
5. API saves data in MongoDB using Mongoose.
6. Admin opens /shield dashboard.
7. Dashboard fetches stored data from the API.
8. Admin updates request status.
9. API saves the new status.
10. Dashboard displays the updated information.

In short:

Browser -> Next.js Frontend -> API Routes / Express API -> MongoDB -> Admin Dashboard

20. DEPLOYMENT AND ENVIRONMENT
------------------------------

The project can be run locally using:

- npm run dev for the Next.js frontend
- npm run dev:api for the Express backend
- npm run dev:all to run frontend and backend together
- npm run build to create a production build

Important environment variables:

- PORT
  The backend server port. Default is 4000.

- MONGODB_URI
  MongoDB connection string.

- CLIENT_ORIGIN
  Allowed frontend origin for CORS when using the Express server.

- NEXT_PUBLIC_API_URL
  Optional external API base URL used by the frontend.

If NEXT_PUBLIC_API_URL is not set, the frontend uses the same-origin Next.js API routes.
